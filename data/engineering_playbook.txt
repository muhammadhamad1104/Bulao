01 · SYSTEM OVERVIEW

What you're actually building.
Bulao is a voice-first, Urdu-native AI service orchestrator for Pakistan's informal economy. A
user opens the app, presses a big button, speaks in Urdu, Roman Urdu, or English, and
within ninety seconds has watched six AI agents extract their intent (with a confidence
score and complexity tag), find providers nearby, rank them across six factors with visible
reasoning, produce a transparent price breakdown, book a slot, track service through to
completion, and — if anything goes wrong — resolve the dispute fairly. The magic is that the
agents are visible on screen as they work. That visibility is the agentic moneyshot the rubric
rewards.
WHAT'S NEW IN v2.0

This version of the playbook is rebuilt against the full official Challenge 2 specification (released after v1
was drafted). The architecture grows from five agents to six with the addition of a dedicated Pricing
Agent. Significant new scope: (a) confidence scoring and clarification questions on the Intent Agent; (b)
job-complexity classification (basic / intermediate / complex); (c) a six-factor matching algorithm on the
Ranking Agent; (d) dynamic pricing with a transparent line-item breakdown; (e) a service-quality loop with
en-route tracking and completion checklist; (f) a dispute & escalation workflow with classified resolutions;
(g) scheduling intelligence (conflict prevention, alternate slots).

The system at a glance

Three layers: the Flutter mobile app captures voice and renders the agent-card UI; a Python backend on Cloud Run runs the
six-agent pipeline using Google ADK and Gemini; Antigravity is the IDE that builds the entire stack — not part of the runtime,
but the source of your 25%-weighted Agent Trace deliverable.

03 · THE ANTIGRAVITY WORKFLOW

How to use the IDE so you actually score the
25%.
Antigravity is not just a place to write code. It's an agent-orchestration platform where
multiple AI agents work in parallel on different parts of your codebase, producing verifiable
Artifacts — plans, diffs, screenshots, browser recordings. Most teams will use it like a fancy
autocomplete and lose two-thirds of the 25%. Use it the way it was designed and you bank
those points.

Two views, two modes of work
Editor View
A VS Code-style IDE with an agent sidebar. Use this when you're hands-on with a single file — tab
completions, inline commands, refactors. This is where you go when you need to read the code, not when
you need to build a feature.

Manager View
The agent-first surface. You describe what you want at a high level, Antigravity spawns an agent in its own
workspace, the agent produces a Plan Artifact, you approve or refine it, the agent executes — writing
code, running terminal commands, testing in a browser. You can spawn multiple agents in parallel, each
on a different workspace. This is where the 25%-weighted screenshots come from. Live in Manager View
this week.

■ One Plan Artifact from a non-trivial feature — ideally something where the agent broke down a vague
request into a clear step plan.
■ One browser recording of an Antigravity agent testing the running app. The agent literally drives the
browser; that's the headline capability of the platform.
■ One diff or code review screenshot showing an Artifact the agent produced and a teammate's
comment guiding a revision.
■ One Knowledge Base entry showing useful context being saved for future agents — a system
prompt, a design token list, an API contract.

Three prompting patterns that work
Pattern A: the parallel kickoff
On Day 1, open three Manager View workspaces and paste a different scaffolding prompt in each. All
three agents run simultaneously. By the time you've finished your coffee, you have a working Flutter app, a
running FastAPI backend, and a deployed Cloud Run service.

Pattern B: the iterative refinement
Don't try to one-shot complex features. Start with: "Create a stub Intent Agent that always returns the same
JSON." Verify it runs. Then: "Now make it actually parse the user input using Gemini Flash with this
system prompt: ..." Verify again. Then: "Add Urdu/Roman Urdu test cases to tests/intent_agent_test.py
and make sure they pass." Three small wins beat one big stuck.

Pattern C: the browser-verified handoff
Ask the agent to verify the work in the browser. "After implementing the AgentCard widget, run the Flutter
app, navigate to the home screen, trigger a sample intent, and take screenshots showing the card
animating through pending → active → done states." The browser recording of this is one of your best
Artifacts.

Think of Antigravity as four junior engineers working for you in parallel. Your job is to give clear
briefs, review Plan Artifacts before they code, and document the good work for the submission
package. The 25% rubric weight is yours to lose.

PROMPT B · BACKEND TRACK · PASTE IN ANTIGRAVITY MANAGER VIEW

Create a Python backend at backend/ that hosts the 5-agent system for Bulao. Use Python 3.11.
1. Create a virtual environment at backend/.venv. Add a requirements.txt with: google-adk, fastapi,
uvicorn[standard], google-genai, pydantic, python-multipart, firebase-admin, pillow, reportlab, python-dotenv.
2. Create backend/main.py — a FastAPI app exposing:
- POST /orchestrate — body {"text": str, "user_location": Optional[Dict[str, float]]}, returns the full pipeline
result with each agent's output.
- POST /book — body {"booking_request": {...}}, returns {"booking_id": str, "receipt_url": str,
"confirmation_message": str}.
- GET /status/{booking_id} — returns the booking record.
- GET /health — returns {"status": "ok", "version": "0.1.0"} for Cloud Run health checks.
3. Set up the ADK agent skeleton at backend/agents/. Create six files, each with a placeholder that returns
hardcoded JSON. We will replace placeholders one agent per day.
- intent_agent.py — defines an ADK LlmAgent named "intent_agent" using model "gemini-flash-latest".
System prompt is just "Return the user input as JSON with a 'service_type' field for now."
- discovery_agent.py — a plain Python function discover(intent: dict) -> List[Dict] that loads
backend/data/providers.json and returns the first 5 entries for now.
- ranking_agent.py — an ADK LlmAgent named "ranking_agent" using "gemini-3-pro-latest". Placeholder
prompt: "Pick the first provider and explain why in one sentence."
- pricing_agent.py — an ADK LlmAgent named "pricing_agent" using "gemini-flash-latest". Placeholder:
returns a stub price breakdown with base_visit_fee, distance_fee, and total.
- booking_agent.py — a plain Python function book(provider_id: str, intent: dict, price: dict) -> Dict that
returns a stub booking record including service-quality fields (status: "confirmed", en_route_at: null,
in_progress_at: null, completed_at: null).
- followup_agent.py — a plain Python function with two entry points: schedule_followup(booking_id) for
reminders/check-ins, and handle_dispute(booking_id, dispute_type, user_message) for the dispute path.
4. Create backend/agents/orchestrator.py that sequences the six agents. For Day 1, use plain Python
function calls in sequence. Day 2 we'll convert to ADK's SequentialAgent.
5. Create backend/data/providers.json with 100 fake Pakistani service providers. Each provider object has: id
(string, "prov_001" to "prov_100"), name (Pakistani Urdu/English names), service_categories (array of
strings from this list: "plumber", "electrician", "ac_technician", "geyser_technician", "carpenter", "painter",
"beautician", "tutor", "appliance_repair", "gas_leak_specialist"), specializations (array of sub-skills, e.g.
["inverter_ac", "split_ac"] for AC techs, ["bridal", "party"] for beauticians), lat (decimal between 33.4 and 33.7),
lng (decimal between 72.9 and 73.2), neighborhood (F-6, F-7, F-8, F-10, F-11, G-7, G-9, G-10, G-11, G-13,
Bahria Town, DHA, Saddar, Pindi Cantt), rating (decimal 3.5–5.0), completed_jobs (integer 5–200),
completed_jobs_by_neighborhood (object mapping neighborhood name to count), on_time_score (decimal
0.5–1.0; reliability metric), cancellation_rate (decimal 0.0–0.3), review_recency_days (integer 1–180 since
most recent review), risk_score (decimal 0.0–1.0; computed from cancellations + complaints; lower is better),
current_workload (decimal 0.0–1.0; how booked-up they are right now), available_slots (array of ISO
datetimes for the next 7 days), base_visit_fee_pkr (integer 300–800), rate_per_hour_pkr (integer 500–2000),
phone ("+923XX-XXXXXXX" format), gender ("male" or "female"), years_experience (integer 1–25),
languages (subset of ["urdu", "english", "punjabi"]), verified (boolean), strikes (integer 0–3; dispute history).
The richer schema supports six-factor matching, dynamic pricing, and dispute handling — do not skip any
fields.
6. Create a Dockerfile that uses python:3.11-slim, copies requirements, installs them, copies the app, and
runs uvicorn main:app --host 0.0.0.0 --port 8080. Cloud Run sets the PORT env variable; respect it.

Morning · the 50-example utterance dataset
Before touching code, the Product/Urdu lead spends ninety minutes drafting fifty real Pakistani utterances.
This dataset is the source of truth for prompt tuning, the regression suite for the Intent Agent, and the basis

3. Pricing logic (deterministic, computed in Python BEFORE asking the LLM to write the explanation):
- base_visit_fee = provider.base_visit_fee_pkr
- estimated_hours by complexity: basic 1.0, intermediate 1.5, complex 2.5
- estimated_service_fee = provider.rate_per_hour_pkr * estimated_hours
- distance_fee = max(0, distance_km - 3) * 50 PKR (3km free)

- with_urgency = subtotal + urgency_surcharge
- with_surge = with_urgency * surge_factor
- total = with_surge + loyalty_discount
- Round to nearest 10 PKR
- range = (total * 0.9, total * 1.15); ensure range is at least 250 PKR wide

APPENDIX D

Failure modes & final submission checklist.
Hackathon teams lose more often to recurring failure patterns than to lack of talent. Here are
the ten that kill projects, ranked by frequency, and the pre-submission checklist that catches
them before they cost you points.

The ten failure modes
01

Faked Urdu support. Mobile UI is in Urdu but the agent only really handles English. Judges will
speak Urdu during finals; the agent will fall over. Fix: the 50-example dataset and the Intent Agent
prompt iteration. Aim for 92%+ accuracy on the dataset.

02

Invisible agents. The system uses agents but the UI doesn't show them reasoning. The
20%-weighted Agentic Reasoning score evaporates. Fix: the AgentCard UI with deliberate reveal
timing (Section 06).

03

Antigravity trace as afterthought. Team scrambles to take screenshots on Day 5; trace looks thin
and forced. Fix: the daily screenshot checklist in Section 03. Start Day 1, not Day 5.

04

Demo runs on a simulator. Judges can tell. The demo loses credibility. Fix: film on a real Android
device. Charge it the night before.

05

Voice fails mid-demo. Live voice fails at finals; team panics. Fix: if voice isn't reliable on three out of
three test runs on Day 4, ship text-only. Add DEMO_MODE fallback (Appendix C).

06

Ugly receipt PDF. The Booking Agent's receipt looks like a default ReportLab dump. Judges see it
on screen for 2 seconds in the demo. Fix: design the receipt deliberately. Match the Bulao design
tokens. Iterate until it could pass for Stripe.

07

Ranking reasoning sounds generic. "Best option for you" instead of "Only 1.4km away, 4.8 rating,
47 jobs in your area." The agent-reasoning score takes a hit. Fix: the Ranking Agent prompt and its
must-contain-a-digit validator.

08

README is a placeholder. Judges open the repo, see TODO sections, mark down on the
documentation/polish score. Fix: the README structure on Day 5 morning, fully filled.

09

Cloud Run service is down at finals. Min-instances was set to 0 and the cold-start kills the demo.
Fix: set min-instances to 1 for the finals presentation window; revert after.

10

Single point of failure for screen-recording. Only one teammate has the recording, their laptop
